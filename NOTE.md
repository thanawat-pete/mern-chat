เอกสารฉบับนี้เป็นบันทึก (Note) สำหรับปูพื้นฐาน **Object-Oriented Programming (OOP)** หรือ "การเขียนโปรแกรมเชิงวัตถุ" และหลักการเขียนโค้ดที่ดี (Clean Code) เหมาะสำหรับใช้อ่านทบทวนก่อนสอบหรือก่อนเริ่มเขียนโปรเจกต์ขนาดใหญ่

## 🏗️ 1. OOP คืออะไร? (What is OOP?)
สมัยก่อนเราเขียนโค้ดแบบเรียงจากบนลงล่างตามลำดับ (Procedural) ทำให้อ่านยากและแก้ไขโค้ดยากมากเมื่อโปรเจกต์ใหญ่ขึ้น

**OOP (การเขียนโปรแกรมเชิงวัตถุ)** จึงถือกำเนิดขึ้น โดยมีแนวคิดว่า **"มองทุกอย่างในโปรแกรมให้เป็นสิ่งของ (Object)"** เพราะโลกความจริงเราก็มองทุกอย่างเป็นสิ่งของ 

### คำศัพท์พื้นฐานที่สุดที่ต้องรู้:
1. **Class (คลาส / พิมพ์เขียว)**
   - คลาสคือ **"แบบแปลน"** หรือแม่พิมพ์ที่เอาไว้สร้างสิ่งของ
   - ตัวอย่าง: คลาส `Car` (แบบแปลนรถยนต์) ระบุว่ารถทุกคันต้องมียี่ห้อ (Brand) มีสี (Color) และสามารถสตาร์ทเครื่องได้ (Start Engine)
2. **Object / Instance (อ็อบเจกต์ / วัตถุจริง)**
   - อ็อบเจกต์คือ **"ของจริง"** ที่ถูกสร้างออกมาจากแม่พิมพ์ (Class)
   - ตัวอย่าง: เราเอาคลาส `Car` ไปสร้าง `myCar` (รถของฉันยี่ห้อ Honda สีแดง) และ `friendCar` (รถของเพื่อนยี่ห้อ Toyota สีดำ) 
   - `myCar` และ `friendCar` คือ Object (ของจริง) ที่สร้างมาจาก Class `Car`

## 🏛️ 2. เสาหลัก 4 ประการของ OOP (The 4 Pillars of OOP)
ข้อสอบมักจะออก 4 หัวข้อนี้เสมอ ต้องจำให้แม่น!

### 1. Encapsulation (การห่อหุ้ม)
- **คอนเซ็ปต์:** เอาตัวแปร (Data) และฟังก์ชัน (Methods) มามัดรวมกันไว้ในแคปซูล (Class) เดียวกัน และ "ซ่อน" สิ่งที่ไม่อยากให้คนภายนอกแก้ได้ตามใจชอบ
- **ตัวอย่างเปรียบเทียบ:** รีโมททีวี แผงวงจรข้างในถูกห่อหุ้มไว้ (ซ่อนไว้เป็น Private) เราในฐานะผู้ใช้งานมีสิทธิ์แค่กดปุ่ม (Public Method) เปลี่ยนช่องเท่านั้น เราไม่มีสิทธิ์แกะรีโมทไปจี้สายไฟเอง
- **ประโยชน์:** ป้องกันไม่ให้โค้ดส่วนอื่นมามั่วข้อมูลของเราจนบัคกิน

### 2. Abstraction (นามธรรม)
- **คอนเซ็ปต์:** ดึงมาแค่ข้อมูลหรือฟังก์ชัน "ที่จำเป็นต่อการใช้งานจริงๆ" โดยซ่อนความซับซ้อนเบื้องหลังการทำงานเอาไว้ (Show only essential features, hide background details)
- **ตัวอย่างเปรียบเทียบ:** การขับรถยนต์ เราแค่รู้ว่าเหยียบคันเร่งแล้วรถจะวิ่งไปข้างหน้า (ใช้งานผ่าน Abstraction) โดยที่เรา **ไม่จำเป็นต้องรู้** ว่าพอกดคันเร่งแล้ว เครื่องยนต์สูบฉีดน้ำมันอย่างไร สายพานหมุนแบบไหน
- **ประโยชน์:** ใช้งานง่าย ลดความซับซ้อน (ซ่อน Logic ยากๆ ไว้ข้างหลัง)

### 3. Inheritance (การสืบทอด)
- **คอนเซ็ปต์:** การสร้าง Class ใหม่ โดยรับเอาคุณสมบัติ (Variables) และความสามารถ (Methods) มาจาก Class แม่ ทำให้ไม่ต้องเขียนโค้ดซ้ำ
- **ตัวอย่างเปรียบเทียบ:** ไก่ (Chicken) และ นกกระจอกเทศ (Ostrich) สามารถสืบทอดมาจากคลาสแม่ชื่อ นก (Bird) ทำให้ทั้งไก่และนกกระจอกเทศมี "ปีก" และ "จะงอยปาก" โดยอัตโนมัติ ไม่ต้องเหนื่อยเขียนใหม่
- **คำศัพท์:** คลาสแม่เรียกว่า Superclass / Parent, คลาสลูกเรียกว่า Subclass / Child

### 4. Polymorphism (การพ้องรูป / ความหลากหลาย)
- **คอนเซ็ปต์:** Class หลายๆ ตัวสามารถมี "ชื่อฟังก์ชันเดียวกัน" แต่ทำงานโค้ดข้างใน "ไม่เหมือนกัน" ได้ (Override)
- **ตัวอย่างเปรียบเทียบ:** คลาสแมว นก และหมา ล้วนสืบทอดมาจากคลาส "สัตว์" (Animal) 
  - สัตว์ทุกตัวมีฟังก์ชัน `makeSound()` (ส่งเสียงร้อง)
  - หมา เรียก `makeSound()` แล้วเห่าโฮ่งๆ
  - แมว เรียก `makeSound()` แล้วร้องเหมียวๆ
  - การที่ฟังก์ชันชื่อเดียวกัน แต่แสดงผลต่างกันตามบริบทของ Object นั้นๆ คือ Polymorphism!

## 🚀 3. OOP ใน JavaScript (ภาษาที่ใช้ใน MERN)
JS เป็นภาษาที่แปลกนิดนึง เพราะพื้นฐานเดิมมันไม่ได้เป็น OOP แบบ 100% เหมือนพวก Java หรือ C++ แต่มันเป็น **Prototype-based OOP**

- แต่ตั้งแต่ปี 2015 (ES6) ภาษา JS ได้เพิ่มคำสั่ง `class`, `constructor`, `extends` เข้ามา ทำให้เขียนเหมือน OOP ปกติได้แล้ว (แต่เบื้องหลังก็ยังแอบเป็น Prototype อยู่ดี)
- **การประยุกต์ใช้ในโปรเจกต์จริง:**
  - ไฟล์ `message.model.js` ของ **Mongoose** คือตัวอย่างของการทำ OOP เราประกาศ Schema (คล้าย Class) แล้วตอนใช้งานเราเอามาดึงหาข้อมูล `Message.find()`
  - ฝั่ง **React** สมัยใหม่ (Function Component + Hooks) มักจะหลีกเลี่ยงการทำ "Inheritance" แบบคลาสสิก แต่เน้นใช้หลักการ **Composition** (เอาคอมโพเนนต์เล็กๆ มาประกอบร่างเป็นอันใหญ่) แทน เพราะยืดหยุ่นกว่ามาก

## 💎 4. กฎเหล็ก 3 ข้อ เพื่อการเขียนโค้ดที่สะอาด (Clean Code Basics)
นอกจากรู้ OOP แล้ว ทฤษฎีการเขียนโปรแกรมที่ดีจะทำให้อาจารย์ให้คะแนนความประทับใจเพิ่มขึ้น:

### ข้อที่ 1: การตั้งชื่อ (Naming Convention) ตัวแปรต้องเล่าเรื่องได้
- **อย่าทำ:** `let x = 10;`, `function fn1() {}`
- **ควรทำ:** `let maxUsers = 10;`, `function getUserProfile() {}`
- ชื่อฟังก์ชันควรเป็น **กริยา (Verb)** เสมอ (เช่น `sendMessage`, `getUsers`)
- ชื่อตัวแปรควรเป็น **คำนาม (Noun)** (เช่น `userList`, `email`)

### ข้อที่ 2: กฎ DRY (Don't Repeat Yourself)
- อย่าก๊อปปี้โค้ดไปแปะซ้ำๆ 
- ถ้าเห็นโค้ดไหนหน้าตาคล้ายกันเกิน 2 จุดในระบบ ให้จับมันโยนใส่ฟังก์ชันกลาง หรือทำเป็น Component แยก แล้วค่อยเรียกใช้แทน ถ้าต้องแก้บัค จะได้แก้แค่ที่เดียวไม่ต้องไล่แก้ทุกไฟล์

### ข้อที่ 3: กฎ SRP (Single Responsibility Principle)
- 1 ฟังก์ชัน หรือ 1 ไฟล์ ควรมีหน้าที่ในการทำงาน **"เพียงอย่างเดียวเท่านั้น"** 
- ตัวอย่าง: ฟังก์ชัน `loginUser()` ก็ควรมีแค่ Logic การเข้าสู่ระบบ ไม่ควรแอบมีโค้ดต่อ Socket ดึงข้อความแชทพ่วงปนมาด้วย ถ้าพยายามจะทำหลายอย่าง ให้แยกเป็นฟังก์ชันใหม่
- นี่คือเหตุผลที่โครงสร้างโปรเจกต์ MERN ต้องแยกโฟลเดอร์ `controllers` (คิดคำนวณ), `models` (คุยกับคลัสเตอร์ DB), `routes` (แยกเส้นทาง) ออกจากกันอย่างเจาะจง!
